# Data Model

The same shapes are implemented three times — as Java POJOs (`android-app`), as plain objects validated in `backend-api`, and as Firestore documents — so this page is the single source of truth all three implementations are checked against.

## Firestore layout

```
users/{uid}
  ├─ (fields as above)
  ├─ plans/current              # latest WorkoutPlan (also mirrored to backend for generation)
  └─ workout_logs/{logId}       # one doc per completed session
```

## Firestore Security Rules (summary)

```
match /users/{uid} {
  allow read: if request.auth.uid == uid;
  allow write: if false; // profile fields are only written by backend-api (Admin SDK)

  match /plans/{planId} {
    allow read: if request.auth.uid == uid;
    allow write: if false; // plans are generated by backend-api
  }

  match /workout_logs/{logId} {
    allow read: if request.auth.uid == uid;
    allow write: if false; // logs are written by backend-api after validation
  }
}
```

Cross-user reads (the trainer roster) are **never** exposed to Firestore Security Rules directly — they only exist behind `GET /api/trainer/roster`, which runs with the Admin SDK and checks the `trainer` custom claim server-side. This is the one deliberate asymmetry in the architecture: trainees get real-time direct reads for the fast/offline single-user case, trainers always go through the API for the cross-user case.
